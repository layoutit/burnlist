title: The .oven system
description: How declarative Oven source becomes a rendered detail page.
---

A `.oven` file is declarative, non-executable detail-page data. Its XML-like `<oven id version contract theme refresh-seconds>` root names the Oven, its data contract, its theme, and an optional refresh interval:

```xml
<oven id="streaming-diff"
      version="0.1.0"
      contract="burnlist-streaming-diff-data@2"
      theme="streaming-diff">
  <streaming-diff-heading session="/identity/session" back-href="/backHref"/>
  <diff-card source="/cards"/>
</oven>
```

This is the complete committed `streaming-diff.oven`. It describes what to render; it cannot run commands, evaluate code, import UI, or inject a renderer.

The `version` is the Oven's semver `id@version` identity, distinct from its content revision. A shipped Oven can be pinned per project with `burnlist oven adopt <id>` under committed `.burnlist/ovens/<id>/`; use opt-in `burnlist oven upgrade <id>` to move that pin.

## Rendering pipeline

```text
<id>.oven (human source)
  -> compileOven
  -> build-generated IR
  -> OvenRuntime with the selected theme
  -> rendered detail page
```

The build's oven-ir Vite plugin compiles `<id>.oven` at build or import time. `compileOven(source)` validates the source against the DSL grammar and emits JSON-only IR; the shared `OvenRuntime` renders it with the selected theme from the theme registry. The IR is never a committed file. The result is composable detail-page UI without executable Oven content.

## Minimal footprint

Every Oven directory contains its two canonical files. It has a flat data layer
only when it needs a custom data source:

```text
<oven-id>/
  instructions.md     # outcome, state, inputs, and evidence rules
  <id>.oven           # declarative detail-page source
  contract.mjs / handler.mjs / adapter / data.schema.json  # optional, flat data layer
```

The oven-ir Vite plugin generates IR from `<id>.oven`; authors never create or commit `.ir.json`. Checklist has no data layer. Performance Tracing and `visual-parity` have flat `contract.mjs` and `handler.mjs` files; Differential Testing has a flat data layer of `contract.mjs`, `handler.mjs`, `data-contract.mjs`, `adapter-sdk.mjs`, `transport.mjs`, `worker-runtime.mjs`, and `data.schema.json`. Streaming Diff is the one Oven that keeps a large `engine/` directory and `e2e/`.

## Creating an oven

`New Oven` takes `{id, name, instructions}` and the server scaffolds a starter
`.oven`. The former drag-grid builder was removed.

## Why use a DSL

- **Composable:** the grammar combines registered layout, display, binding, and control elements.
- **Safe:** Ovens ship data, not code. The compiler accepts a closed vocabulary and the runtime owns rendering.
- **Consumer-authorable:** a project can author a declarative view over its normalized data without importing project UI into the dashboard.

See the [.oven DSL reference](/ovens/dsl-reference) for the complete vocabulary, or [create your own Oven](/ovens/authoring) for the authoring path.
